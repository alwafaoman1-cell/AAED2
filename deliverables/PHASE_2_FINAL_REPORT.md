# تقرير Phase 2 النهائي

التاريخ: 22 يونيو 2026

## النتيجة

- أصبحت البيانات التشغيلية المستهدفة تعتمد على Supabase كمصدر الحقيقة: المطالبات، أوامر العمل، المصروفات، العملاء، المركبات، الفواتير، التقارير، الأرشيفات، وسجل واتساب.
- أزيلت بيانات العرض التلقائية وملفات الـ demo، ولا يبدأ النظام بسجلات تجريبية.
- الطباعة والمعاينة والتنزيل أصبحت مبنية على ملف PDF واحد؛ لا تُستخدم طباعة HTML في مسارات الطباعة التشغيلية.
- إرسال واتساب من الواجهة يمر عبر Edge Function `whatsapp-meta-send`، دون وضع token أو secret في كود الواجهة.
- أضيف سجل `whatsapp_logs` مرتبط بالمستأجر والعميل والمركبة والمطالبة وأمر العمل، مع RLS وRealtime.
- أضيفت مزامنة قاعدة البيانات بين أمر العمل والمطالبة، وتحديث فوري لأمر العمل عند تغيير مصروف مرتبط.
- أضيفت عروض تقارير Supabase لعمليات الورشة، السيارات المسلمة، كشف التأمين، أرشيف المطالبات، وأرشيف الفواتير.
- أزيل تجاوز التتبع العام بكلمة مرور محلية، وأصبح التحقق عبر RPC في الخادم فقط.

## Migration الجديدة

`supabase/migrations/20260622090000_phase2_cloud_source_whatsapp_constraints.sql`

تتضمن:

- فحص التكرارات أولاً والتوقف الآمن عند وجودها.
- قيود عدم التكرار المطلوبة للمطالبات وأوامر العمل وVIN غير الفارغ.
- جدول `whatsapp_logs` وسياسات RLS.
- Triggers للمزامنة.
- Views للتقارير والأرشيفات.
- تفعيل Realtime للجداول التشغيلية ذات الصلة.

مهم: أُضيفت الـ migration إلى المشروع ولم تُطبّق على قاعدة Supabase البعيدة، لأن بيئة العمل لا تحتوي Supabase access token أو بيانات اتصال قاعدة البيانات. يجب تشغيل `supabase db push` من حساب مخوّل. ستتوقف الـ migration تلقائياً وتعرض خطأ إذا وجدت تكرارات، ولن تطبق قيداً خطراً فوق بيانات مكررة.

## الأمان

- سياسات `whatsapp_logs` مقيدة بالمستأجر والأدوار.
- أسرار Meta WhatsApp تُقرأ داخل Edge Function فقط.
- روابط التتبع العام لا تستخدم بيانات محلية أو كلمة مرور رئيسية داخل المتصفح.
- روابط الملفات العامة الحساسة تعتمد على آليات التوقيع الموجودة ومسارات tenant.
- لم تُضمّن ملفات `.env` أو أسرار في ملف ZIP النهائي.
